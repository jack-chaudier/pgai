# Architecture

I chose Nova Sonic for the patient bot because I've used it before and a speech-to-speech model is a natural fit for this task. The bot role-plays as a patient -- it doesn't need to look up data or do multi-step reasoning, it just needs to listen and respond naturally. A traditional STT → LLM → TTS pipeline (e.g., Whisper + GPT + ElevenLabs) would make sense if the bot needed to query external systems or do analysis mid-call, but for pure conversational role-play, a single bidirectional stream gives lower latency and no orchestration overhead. Twilio was the obvious choice for telephony -- it has the most straightforward API for outbound calls with media streams. FastAPI over Flask because it has native WebSocket and async support, which Twilio's media streams require. Ngrok to expose the local server since Twilio needs a public webhook URL.

Nova Sonic runs in a dedicated background thread with its own `asyncio` event loop because the `awscrt` HTTP/2 client that powers the Bedrock SDK conflicts with uvicorn's event loop when initialized in the same context. The bridge communicates across threads through `queue.Queue` objects -- one for inbound audio, one for outbound. This two-thread design was arrived at empirically: attempts to run both in a single event loop produced silent failures. Audio conversion between Twilio's 8kHz mu-law and Nova Sonic's 16/24kHz PCM is handled by `scipy.signal.resample_poly` -- naive interpolation would alias when downsampling, producing audible artifacts.

Post-call analysis uses Gemini 3.1 Flash Lite because Nova Sonic was already the most expensive component and the analysis step is just text classification -- reading transcripts and flagging bugs. A cheaper, smaller model keeps costs down without sacrificing quality on a task that has no audio or real-time constraints. The batch runner drives all 16 scenarios sequentially: set the scenario via API, place the Twilio call, wait for completion, save the transcript, move to the next.
