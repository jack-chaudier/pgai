# Architecture

The system is a voice bot that calls a target phone number via Twilio and role-plays as a patient using Amazon Nova Sonic for speech-to-speech AI. A FastAPI server (`server.py`) accepts incoming Twilio media stream WebSockets and bridges them to Nova Sonic through a `TwilioNovaBridge`. We chose Nova Sonic's bidirectional streaming over a traditional STT + LLM + TTS pipeline because it collapses the entire voice interaction into a single HTTP/2 stream with lower latency and no orchestration between separate services. Audio conversion between Twilio's 8kHz mu-law and Nova Sonic's 16kHz/24kHz PCM formats is handled by `audio.py`, which uses `scipy.signal.resample_poly` for sample rate conversion -- simple linear interpolation would alias when downsampling 24kHz to 8kHz, producing audible artifacts.

Nova Sonic runs in a dedicated background thread with its own `asyncio` event loop because the `awscrt` HTTP/2 client that powers the Bedrock SDK conflicts with uvicorn's event loop when initialized in the same context. The Bedrock client is created at import time, before uvicorn starts, and the bridge communicates with it through thread-safe `queue.Queue` objects -- the inbound queue carries caller audio to Nova, the outbound queue carries Nova's responses back. This two-thread, two-loop design was arrived at empirically: attempts to run both in a single event loop produced silent failures.

After each call, the bridge saves a transcript (JSON + readable text) and a stereo WAV recording. An optional post-call analyzer (`analyzer.py`) sends transcripts to Gemini 3.1 Flash Lite, chosen because transcript analysis is a pure text task where a cheaper, smaller model is sufficient. A batch runner drives multiple scenarios end-to-end: set scenario via API, place Twilio call, record transcript, analyze for bugs.
